# AI Capability Report v0.1 — LVGL on GD32F303

**Period**: 2026-05-01 立项 → 2026-05-02 sd 完成（约 1.5 天）
**Scope**: Card 1（最小可闭环调试系统）+ Card 2（LVGL 表盘）共 8 sub-task
**Methodology**: B 路径双轨记录（每 sub-task pre/post 自评 + 周 retro）

## 一、分层完成度（AI 独立完成 vs 用户介入）

| 层 | 子任务 | 用户介入次数 | AI 独立完成度 | 失败模式 |
|---|---|---|---|---|
| **工程 infra** | st1 (pre/post 工具 + smoke gate) | 0 | 100% | 跨平台 shell quoting / git --since UTC（自查自修） |
| **bring-up** | st2 (CMake + GCC startup + linker) | 0 | 100% | GD32F30X_HD 宏漏定义（一次 build 错暴露后修复） |
| **烧录链路** | st3 (pyOCD + ST-Link) | 1（用户告知 ST-Link 型号） | 95% | 无 |
| **GDB 调试** | st4 (gdb-init script) | 0 | 100% | 无（一次跑通） |
| **屏驱动** | sa (ST7789V3 1.47") | 2（用户澄清 IIC→SPI / 1.14→1.47 误判） | 80% | AI 没核对 KIT 实际 example 用的 SPI 通道（references 写错 SPI0 remap，实际 SPI2 默认） |
| **LVGL 移植** | sb (v9.5.0 + flush_cb) | 1（颜色错位用户报） | 70% | RGB565 字节序（3 步硬件诊断后修复）/ size 调试 6 回合（理论上更早能解决） |
| **RTC + sysmon** | sc (LSE + 时间显示) | 0（RTC 部分） + 4（sysmon 加时撞车 / banner 残影 4 次错诊断） | 60% | sysmon 依赖 LV_USE_SYSMON 漏开 / spinner 依赖 LV_DRAW_SW_COMPLEX 漏开 / banner 残影歧义未澄清 |
| **GUI 设计** | sd (GT3 watchface) | 5（半屏 / 半屏 / 又半屏 / 屏尺寸验证 / 最终确认） | 50% | LVGL transform_scale + partial buffer 不兼容（4 次假设全错，未深入到 byte-level 定位）/ 大字字体 Flash 超 256K |

**总用户介入：13 次**（其中 9 次集中在 sb 后期 + sd），其中 4 次属"用户视觉反馈"（v0.2 自动化目标）。

## 二、能力维度评估

### 2.1 强项（AI 表现稳定）

1. **工具链 + 烧录链路**：从 0 到 GDB attach 用 ~30 分钟。CMake / linker / 启动汇编 / OpenOCD vs pyOCD 选型决策 / SWD 连接 — AI 可独立完成。证据：st1-st4 共 0 用户技术介入。

2. **vendor SDK 抄写 + 适配**：KIT/Examples/RGB LED 抄到 src/，KIT/LCD 1.47inch 抄屏驱动，KIT/RTC 抄基础。AI 能识别"哪些是 vendor 代码可抄、哪些是项目特定要适配"。证据：sa/sc 一次跑通。

3. **GDB 自动化诊断**：gdb-init.txt 非交互脚本 + 5 次采样 PC 诊断死循环 + addr2line 翻 PC → 函数名。AI 在 GDB 操作上比手工快 3x（不会输错命令）。

4. **多论据已知 datapoint**：30 FPS @ 120×120 spinner = 92% SPI 利用率，与理论 940 KB/s 物理上限互证。

### 2.2 弱项（AI 容易卡）

1. **LVGL v9.5 配置项隐式依赖**：开 PERF_MONITOR / spinner / transform 都需要 grep 同名宏看附近依赖。AI 倾向"只开当前 flag 不查关联"，反复 build 失败。证据：sb 6 回合 + sd 4 回合 debug，多数源于配置依赖盲区。

2. **歧义自然语言需求处理**：用户说"banner 残影" → AI 猜了 3 次错方向（GRAM 残留 / partial buffer / lv_refr_now），实际是字 AA 边缘视觉。**没问"残影具体长什么样"** = 假设错。

3. **vendor library 深度 bug 定位**：transform_scale + partial buffer 不兼容 4 次实证，但 AI 没有 fully 定位到具体 byte-level 失败点（layer alloc / stride / chunk size 都 grep 验证过）。结论：AI 在多文件 cross-reference 的 layer/buffer/stride 交互链上能力中等。

4. **size 紧约束下的 trade-off 设计**：256K Flash 紧到极限时，AI 选项空间小 + 缺乏"换一种实现路径"的灵活思考（比如转 lv_canvas 自画大字一开始没想到，是 BFS 后才出）。

### 2.3 全屏 FPS 实测（多论据）

| 场景 | dirty area | dirty bytes/frame | FPS | 实测 KB/s | SPI 利用率 |
|---|---|---|---|---|---|
| 局部小区域刷新（120×120 spinner） | 14400 px | 28800 B | 30 | 864 | 92% |
| 整屏理论上限（320×172 全刷） | 55040 px | 110080 B | ~8.5 | 935 | 99% |

SPI2 配置 7.5 MHz / 8-bit polling / 软件 NSS。**结论**：当前架构 FPS 上限主要由 SPI bandwidth bound。如需更高 FPS 需 (a) DMA 而非 polling SPI (b) 提高 SPI 时钟（实测可推到 30+ MHz on ST7789）(c) 减少 dirty area。

## 三、AI 角色定位（v0.1 实测）

> 设计 doc 写："AI 决策 + 实现，用户验收 + retro 提炼。不是 hands-off"

实测：v0.1 用户介入 13 次 + 大量"看屏报亮没亮"。**真正的 hands-off 在 v0.1 期未实现**。

**v0.2 hands-off 北极星**（design.md 已加）：
- 用 GDB 周期读 partial buffer 像素 → hash 校验 → 替代用户视觉
- 用 GDB 自动 halt + PC 异常检测 → 替代用户报"卡住了"
- 验证实验：清空 conversation context，AI 从 design + references + lessons 重做 LVGL-AI-Lab，目标 0 用户介入。

- 事不过三 — 同现象 3 次失败强制重新审视
- 多论据原则 — 性能/容量/正确性 claim ≥ 2 角度互证
- Autonomous Mode + BFS backtrack
- 局部 vs 全局配置纪律
- 嵌入式硬件三步诊断法

# Design: AI + 嵌入式 GUI 开发最佳实践方法论（智能手表 case）

Generated by /office-hours on 2026-05-01

Mode: Builder

## Problem Statement

通过"完全用 AI 构建一个基于 MCU + LVGL 的智能手表"这件具体的事，反向沉淀出一套**"AI + 嵌入式 GUI 全栈开发最佳实践"**方法论。智能手表本身只是 case study 载体——真正交付物是方法论。

最终产出形态是**完整三件套**：
1. **Playbook / Checklist**——prompt 模板、验收 checklist、AI 与人的分工表
2. **工程模板 / 脚手架**——starter repo（CLAUDE.md + prompt 库 + CI + LVGL 适配脚手架），未来起新嵌入式 GUI 项目可直接 fork
3. **AI 能力边界报告**——分层（GUI 布局 / 应用逻辑 / 驱动 / 中断 / 调优）的诚实报告：AI 在哪几层能独立完成，哪几层必须人介入，各层失败模式是什么

## What Makes This Cool

- **元命题**：在嵌入式 GUI 这个传统上"AI 最弱"的领域系统性地测它的能力边界——不是 vibes，是带 retro 素材的诚实报告
- **平台选了 GD32F303RCT6（48KB SRAM、Cortex-M4F、120MHz、无 2D 加速）**——这是个故意的"困难模式"：资料比 STM32/ESP32 少、资源紧、社区例子少。AI 在冷门 + 紧资源平台的能力是个有价值的 datapoint
- **真硬件 bring-up 做种子 case**——选了 AI 传统上最弱的一环作为起点，无论成功失败信息量都最大
- 用户在 Gudsen 做电机控制 STM32H7（参见 user_work_context memory）——这块方法论可直接反哺工作中下一代产品（电机直驱赛车模拟器有 GUI 需求）

## Constraints

- 硬件：手上已有 GD32F303RCT6 开发板 + 1.5"-2.4" 240x240 或 320x240 SPI 屏（无触控）
- 资源限制：48KB SRAM → LVGL 必须 partial buffer（5-10KB），不能整屏 framebuffer
- 时间：5-10 小时/周
- 用户角色定位：AI 写代码 + AI 做技术决策；用户做验收、烧录、retro 提炼。不是 hands-off，但也不参与具体实现决策

## Premises（已 confirmed）

1. 角色：AI 决策 + 实现，用户验收 + retro
2. 方法论价值 > 手表价值——必要时为方法论牺牲 demo 完整度
3. 真硬件 bring-up 是高信息量起点——选这个起点是因为它是 AI 最弱的一环
4. GD32F303RCT6 + 240x240 SPI + 48KB SRAM 在 LVGL 上跑得通（partial buffer 5-10KB 可行）
5. 方法论是 retro 抽出来的，不是预先 SOP——每个 case 后做 retro 提炼章节
6. 资料稀缺是 feature 不是 bug——GD32 资料少本身是"AI 在冷门平台"的方法论维度
7. v0.1 边界：单 case (GD32F303 表盘) 出 Playbook + 部分 capability 章节；脚手架抽象 + 多平台对照留 v0.2+

## Approaches Considered

### Approach A: 极小种子单 case（minimal viable）
单 case；AI 全包；用户只烧录 + 反馈 + retro。4 周出 v0.1。Pros: 最快闭环。Cons: 方法论素材浅。

### Approach B: 双轨记录强方法论（chosen）
case 同 A，但每个 AI 子任务前后强制写自评素材到 `_meta/sessions/`，每周 retro 抽 Playbook。commit 颗粒度细到每 widget。5-6 周出 v0.1。Pros: 素材最厚、Playbook 有结论。Cons: retro 纪律是真成本。

### Approach C: 对照实验版（lateral）
GD32F303 + ESP32-S3 双平台跑同一个表盘做对照实验。7-9 周。Pros: capability report 一上来就有对照数据。Cons: 时间长 + 双线易拖。

## Recommended Approach: **B（双轨记录强方法论）**

理由：
- A 太薄，单 case 单维度，方法论 v0.1 会被认为缺乏代表性
- C 在 5-10h/周节奏下大概率拖到 9 周以后，v0.1 出不来 momentum 会断
- B 是甜点：5-6 周出有素材有结构的 v0.1，后面加 ESP32 case 自然升级到 v0.2（变相成为 C 的渐进式）
- B 最大的复用是用户已有的 superpowers + autonomous-dev-loop + BACKLOG/CHANGELOG/qa-reports/ 流程——retro 节奏可以直接 piggyback 到 autonomous-dev-loop 的 G6 gate

**关键工程约束（B 独有）**：
- AI 每完成一个子任务前**必须**先写 `_meta/sessions/<timestamp>-<task>-pre.md`（任务理解 + 计划 + 卡点预判）
- 完成后**必须**写 `_meta/sessions/<timestamp>-<task>-post.md`（实际遇到 + 修复路径 + 学到的）
- `pre/post` 不全 → smoke-gate 不让 merge（写到项目 CLAUDE.md 里硬约束）
- 用户每周一次 retro：把当周 `_meta/sessions/` 抽成 Playbook 的具体章节
- commit 颗粒度：1 个 LVGL widget = 1 个 commit（最小可逆单元）

## Open Questions

1. **retro 纪律一旦松动方法论素材会断流**——pre/post 不写不让 merge 是硬约束、还是软建议？倾向硬约束（写到 CLAUDE.md，让 smoke-gate 检查）
2. **AI 自评素材颗粒度**：以 commit 为单位 / 以 widget 为单位 / 以子任务为单位？建议第一周先以 widget 为单位试，周末 retro 时 calibrate
3. **第二个 case 是什么** 留到 v0.1 出后决定。候选：(a) 同板加心率/计步 mock 输入扩展应用层；(b) sim 上同表盘做对照；(c) 换 ESP32-S3 做平台对照（=升级到 C）
4. **项目仓库放哪**：personal GitHub 公开 / 私有 / Gudsen GitLab？这决定 Playbook 的写作语气（公开 vs 内部）。建议先 personal GitHub 私有，方法论稳定后再决定开源
5. **smoke-gate 在嵌入式上怎么定义**：UI demo 的 smoke 不像 Web 那样可自动化；候选是"AI 自报跑通 + 用户烧录后录 5s 视频 + 视频成为 qa-report 附件"。需 v0.1 第一周确定

## Success Criteria（v0.1）

完成定义为同时满足：
- ✅ GD32F303 + 240x240 SPI 屏稳定显示静态时间表盘（实机可烧录可看），含 README + CHANGELOG + smoke 视频
- ✅ Playbook v0.1：≥10 条 prompt 模板，≥5 条验收 checklist，≥3 条 AI 失败模式与修复路径
- ✅ Capability Report v0.1：覆盖 ≥3 层（硬件 bring-up / 屏驱动 / LVGL 应用），每层有 AI 成功率 + 1-2 个失败 mini-case
- ✅ `_meta/sessions/` 累计 ≥20 个 pre/post 对——这是后续 case 的素材底座

## What I noticed about how you think

- 你第一句话是"我需要完全通过 AI 构建一个智能手表功能"，但我把这句话拆成 4 个 AI 边界选项（a/b/c/d）问你时，你直接绕开选项答出真正目标"通过这次实践把 AI+嵌入式 GUI 开发的最佳实践整理出来"。这是 reframe 而不是 pick option——说明你对被给出的选项不缺乏怀疑，更像在用对话校准自己的目标
- 选 v0.1 起点时你直接选了 C（真硬件 + 极简表盘），没选 A（sim 最快闭环）。你优先选了**信息量最大、最难的入口**，而不是最容易闭环的——这跟 boil-the-lake "AI 让边际成本几乎为零，就别贪图最容易那条"原则同构
- 平台选了 GD32F303，不是你最熟的 STM32H7（你工作里做电机控制天天用）。你选了**略微离开舒适区**的板子，暗示你看重的是"AI 能不能帮我打到我还没到过的地方"，而不是"AI 能不能加速我已经会做的"——这是工程师在用 AI 时挺难得的取向
- 你 7 条 premise 一次性全过——没有一条要修。要么你确实想清楚了；要么你在用"先放行 + 跑起来再校准"策略。第二种倾向是 b 类完美主义工程师容易缺的，是个好信号
